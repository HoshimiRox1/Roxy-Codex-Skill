# Design Thinking — 视觉化的思维方式

> 这是整个 skill 最重要的一份文档。如果你只读一份 reference,就读这份。

`patterns.md` 给的是 7 种模板,但真实世界的内容远远超过 7 种。你要学会的不是**模板**,是**思维**——看到一段内容,能问自己"这个内容的结构本质是什么",然后选择(甚至创造)最匹配的视觉形式。

---

## ⚠️ 反模式警告:不要默认选阶段卡片

这是过往版本最常见的滑坡。因为阶段卡片模板在 `patterns.md` 里最完整、最容易抄,模型会本能地"先看哪个好抄"而不是"先想内容结构",于是不适合阶段卡片的内容也被塞进阶段卡片里。

## ⚠️ 反模式警告:不要因为"更容易生成"就压制更好的展示形式

另一种常见滑坡是:明明交互、切换、分层展开会更好展示,但因为静态卡片/流程图更快写,就退回静态模板。

这是错的。**Visualize 的重点是"展示"**,不是"选一个最省实现成本的形式"。

当下面这些信号出现时,要认真考虑交互或独立 HTML,而不是默认静态图:

- 内容有多个视角,切换观察比平铺更清楚
- 总览和细节适合分层揭示,一次性塞满会更乱
- 用户虽然没说"做交互",但某种交互形式显然更有展示力
- 内容本身有条件分叉,交互能把"命中 / 未命中""走 A / 走 B"这种分路讲得更清楚
- 内容是多个容易混淆的概念,交互能让读者逐个聚焦,避免术语串线
- dataviewjs 只是 Obsidian 内的实现手段;如果用户没要求 Obsidian,最佳展示形式可以是独立 HTML

快速自检:

**问题 1**:我现在选静态,是因为它最清楚,还是因为它最好写?
- ✅ 最清楚 → 可以继续
- ❌ 最好写 → 停下来重想

**问题 2**:如果允许交互/切换/展开,读者会不会更容易理解?
- ✅ 会 → 优先考虑交互形式
- ❌ 不会 → 静态形式 OK

**问题 3**:我是在受 Obsidian/dataviewjs 的约束,还是在替用户做展示决策?
- 如果用户明确要 Obsidian,遵守 Obsidian 约束
- 如果用户没要求 Obsidian,就选择展示效果最好的介质

**问题 4**:这个内容的最大价值是不是"一眼看完整体"?
- ✅ 是 → 优先保留静态总览,不要拆成逐步点击
- ❌ 不是,重点在分叉判断或概念聚焦 → 可以考虑交互

### 什么时候交互是真增强

最值得交互的,通常不是"更复杂"的内容,而是下面两类:

1. **条件分叉类**
   例如:重连后走全量同步还是部分重同步、命中 backlog 还是未命中、满足条件 A 还是进入路径 B。
   交互的价值是把"为什么走这条路"显性化,而不是让读者自己从一张静态图里推断。

2. **多概念辨析类**
   例如:多个术语、多个角色、多个相近概念之间的区别。
   交互的价值是让读者一次只聚焦一个概念,把"它是什么 / 它不是什么 / 容易和谁混"分开看。

### 什么时候交互是在打断

如果静态图的核心优势是**整体扫视**,交互往往是在拆掉它的长处:

- 完整时序图本来就是为了让人一眼看到整条主线
- 完整流程图本来就是为了先看全貌再回头看节点
- 总览架构图本来就是为了把所有角色同时放进视野

这类内容如果拆成逐步点击,读者会失去对全局的第一眼把握,进入"点一下看一格"的状态。那不是增强,是打断。

**阶段卡片只适合以下特征全部满足的内容:**
1. **有严格先后顺序** — 阶段 2 必须在阶段 1 之后,不能互换
2. **每阶段独立完成特定目标** — 每个阶段都是一个"通关点"
3. **有明确的总进度** — 读者能感知到"走到了哪里、还剩多少"

典型例子:学习路线、施工计划、产品里程碑、考试备考计划、新员工入职流程。

**下面这些内容即使"看起来像阶段",也绝不要用阶段卡片:**

| 内容特征 | 错误理解 | 正确形式 |
|----------|----------|----------|
| 概念之间**并列分工**(比如 Promise / async/await / Generator) | 误以为是演进顺序 | 并列卡片、关系图、维恩图 |
| **循环/迭代**的流程(render → commit → effect → 再 render) | 误以为是线性推进 | 带回流箭头的流程图、状态机 |
| **历史演进**中各时代主题差异大(CSS 发展史) | 误以为是"升级路径" | 水平时间线,让时间轴本身成为视觉主角 |
| 多个并列组件**协作**完成某事(Event Loop 各部件) | 误以为是"步骤" | 空间布局 + 流向箭头,不是堆叠 |
| 多个**同级分类**(Redis 数据结构) | 误以为是"学习顺序" | 知识卡片网格 |

### 快速自检

做视觉化前,问自己这三个问题:

**问题 1**:如果去掉"阶段 1 / 阶段 2 / 阶段 3"这些编号标签,内容之间的顺序还有意义吗?
- ✅ 有意义 → 可以考虑阶段卡片(但仍要过问题 2、3)
- ❌ 没意义 → 换形式

**问题 2**:读者看完**阶段 3**,能说"我已经走到 3/5 了"这种进度感吗?
- ✅ 能 → 适合阶段卡片
- ❌ 不能(比如三者并列没有"进度"概念) → 换形式

**问题 3**:这些"阶段"之间是否有**回流**或**循环**?
- ✅ 有 → 阶段卡片会掩盖循环,换状态机或带回流箭头的图
- ❌ 没有,单向前进 → 阶段卡片 OK

三个问题都通过,才可以用阶段卡片。否则去下面找更合适的形式。

## 核心原则:内容本质决定视觉形式

视觉化是把**抽象结构**翻译成**空间关系**。文字是线性的,视觉是二维的。当内容的结构是"非线性"的——有层级、有并行、有循环、有网状关系——视觉化就能传递文字很难传递的东西。

所以第一步永远是问:**这段内容的结构本质是什么?**

## 七种结构本质(不是模板,是"内容类型")

| 结构本质 | 特征 | 可选视觉形式 |
|----------|------|--------------|
| **序列** | 有先后,沿时间/因果推进 | 时间线、阶段卡片、阶梯、流水线、甘特图 |
| **层次** | 有上下从属关系 | 分层堆栈、树、嵌套框、思维导图 |
| **关系** | 元素间有连接,没有严格层级 | 节点图、概念地图、维恩图、矩阵 |
| **对比** | 多个维度上比较若干对象 | 并列卡片、表格、雷达图、2x2 矩阵 |
| **比例** | 整体中的组成 | 饼图(慎用)、堆叠柱、气泡、树图(treemap) |
| **状态** | 有限状态之间切换 | 状态机、流程图、决策树 |
| **空间** | 元素在空间上有具体位置 | 地图、平面图、座位表 |

选形式的思路应该是:先识别结构本质 → 在该类本质对应的形式中挑 → 根据内容量/密度/交互需求微调。

**举几个例子看怎么思考:**

- "Redis 进阶路线图" → **序列**(阶段推进)→ 阶段卡片
- "React 组件生命周期" → **状态**(有限状态)→ 状态机(也可以是带时间轴的序列)
- "HTTP vs gRPC vs WebSocket" → **对比**(多维度)→ 并列卡片 或 2x2 矩阵(看维度多少)
- "TCP/IP 协议栈" → **层次**(底层支撑上层)→ 分层堆栈
- "React 生态全景图" → **关系**(技术之间引用)→ 节点图 或 概念地图
- "CAP 定理" → **关系**(三选二)→ 维恩图
- "OAuth 2.0 四种授权模式" → **对比** + **序列**(每种是一个流程)→ 并列 4 列,每列内嵌小时序

## 超出 patterns.md 的视觉模式库

patterns.md 给了 7 种最常用模板,但你的"武器库"应该更大。下面是其他值得掌握的模式——不一定每种都写完整 SVG,但理解它们的结构,真要用时你能现场搭:

### 序列类

- **阶梯图**:像楼梯一样往上走,每一级是一个能力里程碑(适合"等级/段位"式的学习进度)
- **水平时间线**:横向推进,上下分叉标注事件/里程碑(适合技术演进史、产品迭代)
- **流水线/管道**:横向的管道,数据/事件从左流向右,每段是一个处理阶段(适合 ETL、CI/CD、编译流程)
- **甘特图**:多任务并行,每个任务是一个时间段条(适合项目计划)
- **日历视图**:按周/月网格排任务(适合学习计划的"每天做什么")

### 层次类

- **协议栈/层堆栈**:横向分层(像 TCP/IP 的 L1→L7),每层一个横条,最底层最宽,越上越窄或等宽
- **嵌套框**:外层包内层,表达"X 属于 Y 属于 Z"
- **组织树**:自上而下的树,每个节点有子节点(目录结构、组织架构)
- **气泡包装(bubble pack)**:大圆内套小圆,表达"X 包含这些"
- **思维导图**:中心主题,向四周放射子主题(适合头脑风暴式的知识梳理)

### 关系类

- **节点图(网络图)**:节点 + 边,边可以有方向(有向图)或权重(带标签)
- **概念地图**:节点 + 边,但边上标注关系动词("导致""依赖""继承")
- **维恩图**:2-3 个圆的重叠区域(适合"共同点 vs 独有点")
- **依赖矩阵**:行列都是组件,格子表示"行依赖列"(适合找循环依赖)
- **泳道图**:多条水平泳道,每道代表一个角色/系统,消息在泳道间穿梭(适合跨角色流程)

### 对比类

- **并列卡片**:2-3 列卡片,每列是一个方案 + 优缺点列表
- **2x2 矩阵**:两个轴,四个象限(adoption 曲线、波特五力、艾森豪威尔矩阵)
- **雷达图**:多维度在一个放射坐标上展开(适合评分类对比)
- **对比柱**:每个维度一组分组柱
- **Before/After**:左右两半,同一事物的两种状态

### 比例类

- **堆叠柱**:一根柱子分段,每段表示一个类别占比
- **树图(treemap)**:矩形分块,大小表示数值,颜色表示分类
- **气泡图**:圆的面积表示数量,位置可以自由
- **饼图**:只在类别 ≤4 且有"显著占优者"时用,否则用堆叠柱

### 状态类

- **状态机**:圆圈表示状态,箭头表示转移,箭头标签是事件
- **决策树**:自上而下分叉,每个节点是一个判断
- **流程图**(带菱形判断框):包含条件分支的过程
- **用户旅程图**:横向时间轴 + 纵向情绪曲线,标注触点和痛点

### 混合与创新

现实内容经常不止一种结构,这时要**混合**:

- 序列 + 对比 → 每个阶段下挂一张对比表
- 层次 + 比例 → 树图(treemap)
- 状态 + 时间 → 泳道序列图
- 关系 + 比例 → 加权节点图(节点大小表示重要度)
- 对比 + 交互 → 带筛选器的卡片网格(dataviewjs)

**不要被模板困住**。如果你看到一段内容,觉得"既不是序列也不是对比,而是……",那就设计一个新形式。好可视化的标准不是"符合某个模板",而是"让读者最快看懂"。

## 视觉语法:每个视觉元素传达什么

一张好图不是"好看",是**每一笔都在说话**。下面是视觉元素的"词性":

### 位置
- 上下/左右的相对位置 → 顺序、因果、层次
- 靠近 / 远离 → 关联度
- 对齐 → 属于同一类
- 错开 → 属于不同类

### 颜色
- **色相**(红绿蓝)→ 分类(不同东西)
- **明度/饱和度** → 强度(同一东西的不同程度)
- **暖/冷** → 情感倾向(暖=正面/活跃,冷=中性/理性)

切忌用色相表示数值(比如"红色 = 大,蓝色 = 小")——这违反直觉,应该用明度。

### 大小
- 圆/块的尺寸 → 数量、权重、重要性
- 注意:面积和长度的感知不同,**柱状用长度**更精准,气泡用面积更美但不精确

### 线
- **实线** → 强连接、主流程
- **虚线** → 弱连接、可选、异步
- **粗细** → 流量、带宽、重要度
- **箭头方向** → 依赖方向、数据流向
- **双向箭头** → 互相影响
- **折线/曲线** → 有弯折,暗示"绕道"或"条件"

**SVG 图层顺序(做架构图/流程图/状态机时最关键的细节之一)**:

SVG 里后画的元素盖在先画的元素上面(相当于 z-index 等于文档顺序)。连线和方块一旦重叠,谁压谁直接决定视觉效果:

- ❌ **先画方块,后画连线** → 线压在方块上,端点戳进方块内部,非常丑
- ✅ **先画连线,后画方块** → 方块把连线端点遮住,视觉上是"线从节点背后接入",干净利落

所以画有连线的结构图(架构图、流程图、状态机、概念图),SVG 元素书写顺序应该是:

1. `<defs>` — marker、gradient 等定义
2. **所有连线** `<line>` / `<path>`(包括虚线、回溯线)
3. **所有节点方块** `<rect>` / `<ellipse>` / `<circle>` / `<polygon>`
4. **节点内部的文本** `<text>`(和节点同级或嵌在 `<g>` 里,顺序跟节点在一起就好)
5. **连线附近的事件标签** `<text>`(和连线同样放在前面即可;如果担心被节点遮,个别标签可以调到后面)

**特例**:时序图里的生命线(贯穿全图的虚线)本身就不和参与者卡片重叠,顺序影响小;但消息箭头应该**最后画**,压过生命线,强调"这是真正的通信"。

**自检方法**:做完图后扫一眼,连线和方块相接处有没有"线钻进方块里一小段"的感觉。如果有,就是图层顺序反了,把连线代码整段挪到方块之前。

### 形状
- **圆角矩形** → 组件/模块(稳态)
- **菱形** → 判断/决策点
- **圆** → 状态/事件
- **圆柱** → 数据库/存储
- **云** → 外部系统/网络
- **剪切角矩形** → 异步/消息队列

不要用形状"炫技",读者看形状已经有预期,违反预期=增加理解成本。

### 留白
- 元素之间的间距传达"分组" — 近的是一组,远的是另一组
- 留白是**信号**,不是浪费

## 常见的可视化陷阱(做了就说明走偏)

| 陷阱 | 问题 | 破解 |
|------|------|------|
| 3D 图表 | 视觉失真,读不准数值 | 永远用 2D |
| 饼图超过 5 片 | 人眼无法比较相似角度 | 用堆叠柱或表格 |
| 彩虹调色 | 无语义,喧宾夺主 | 同类用同色系,对比用互补 |
| 每个元素都描边 + 阴影 + 渐变 | 装饰堆砌 | 去掉阴影和渐变,边框只在必要时 |
| 图例超过 8 项 | 读者记不住 | 分组或直接标注在元素上 |
| 图 + 冗长标题 + 大段说明 + 标注 + 水印 | 信息塞满 | 一张图一个焦点 |
| "为了好看"加图 | 图没承载信息 | 删掉,或者换成能承载信息的图 |
| 用图标代替文字标签 | 图标歧义 | 图标是装饰,标签用文字 |
| 对比多个序列但用不同刻度 | 视觉误导 | 统一刻度(或明确标注) |
| 大面积饱和色 | 视觉刺激,疲劳 | 用浅色背景 + 深色强调 |

## 风格的统一:为什么"像 claude.ai 的 visualize"

用户希望产出的风格和 claude.ai 的 visualize 工具一致。那种风格的几个关键特征:

1. **柔和的粉彩(pastel)背景 + 中等饱和的强调色** — 不锐利,不花哨
2. **透明背景** — 让视觉块"嵌在"笔记里,不是"贴上去"
3. **12px 统一圆角** — 温和不尖锐
4. **极少的阴影** — 几乎完全扁平
5. **系统字体** — 跟随读者环境,不强加字体
6. **信息密度中等** — 不塞满,也不空洞
7. **色系统一** — 一张图里 2-3 个色相,每个色相有明确语义
8. **功能优先** — 所有装饰都让位于信息传递

这套风格之所以好用,是因为它**不抢戏**。笔记的主角是内容和思考,视觉块是帮助理解的工具,不是作品。

**反面:** 博客风、海报风、作品集风、infographic 风。这些风格追求"视觉冲击",不适合深度学习笔记。

## 一张好的"学习笔记视觉块"的 checklist

写完一张 SVG 或做完一个 dataviewjs 块后,对照自检:

- [ ] 去掉这张图,读者理解这段内容会慢多少?(如果差不多,就不该加)
- [ ] 每个颜色能说出它表示什么?
- [ ] 不用看图例,读者能直接看懂每个元素?
- [ ] 去掉任意一个元素,信息量会减少吗?(如果不会,去掉)
- [ ] 在 Obsidian 深色/浅色主题下都能读?
- [ ] 尺寸是否合理(不过宽撑破版面,不过窄显得寒酸)?
- [ ] 如果读者只花 3 秒看这张图,会抓住什么?这正是你想让他抓住的吗?

## 最后:像"老师"一样思考

用户希望你是得意门生,反过来说,**你对用户而言是老师**。一个好老师讲课时会:

- **先讲 why**(为什么要学这个、解决什么问题)再讲 how
- **用类比和具象**把抽象东西变得可触摸
- **留白**让学生有思考空间,不是信息轰炸
- **指出重点**而不是事无巨细都讲
- **承认不确定**而不是强装权威

视觉化就是"类比和具象"的一种形式。你画的每一张图,都应该问:**这是一个学生看了之后会"啊,原来是这样"的图吗**?

如果是,就画。如果不是,不如不画。
